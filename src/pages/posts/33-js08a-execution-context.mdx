## 소스 코드의 타입

ECMAScript에서는 소스코드를 4가지 타입으로 구분하며, 각 타입은 실행 컨텍스트를 생성한다.

- 전역 코드: 전역에 존재하는 코드
  - 최상위 스코프인 전역 스코프를 생성한다
  - 전역 변수와 전역 함수를 전역 객체의 프로퍼티로 바인딩하기 위해 전역 객체와 연결한다
  - 전역 실행 컨텍스트를 생성한다
- 함수 코드: 함수 내부에 존재하는 코드 (중첩 함수를 포함되지 않음)
  - 지역 스코프를 생성한다
  - 지역 변수, 매개변수, arguments 객체를 관리한다.
  - 함수 실행 컨텍스트를 생성한다
- eval 코드: eval 함수의 인수로 전달되는 코드
  - strict mode에서 독자적인 스코프를 생성한다.
  - eval 실행 컨텍스트를 생성한다.
- 모듈 코드: 모듈 내에 존재하는 코드 (모듈 내부 함수, 클래스는 포함되지 않음)
  - 모듈별로 독립적인 모듈 스코프를 생성한다.
  - 모듈 실행 컨텍스트를 생성한다.

## 소스코드의 평가와 실행

자바스크립트의 소스코드는 실행에 앞서 준비 과정을 거친다. 이를 “**소스코드의 평가**” 과정이라고 한다. 그 후 실제로 코드가 실행되는 과정을 “소스코드의 실행" 과정이라고 한다.

- 소스코드의 평가
  - 실행 컨텍스트 생성
  - 변수, 함수 선언문을 먼저 실행하여 스코프에 식별자를 키로 등록
- 소스코드의 실행
  - 선언문을 제외한 코드를 순차적으로 실행
  - 런타임 시작
  - 필요한 정보는 스코프에서 검색하여 취득
  - 소스코드 실행 결과도 스코프에 등록

## 실행 컨텍스트의 역할

```jsx
const x = 2;

function foo(a) {
  const x = 3;
  console.log(a);
}

foo(123);

console.log(1234);
```

위 코드를 실행하는 순서는 아래와 같다.

 1. 전역 코드 평가 단계

런타임 이전에 소스코드의 평가 단계를 거친다. 평가 단계에서는 선언문만 실행한다. 즉, 함수 선언문과 변수 선언문만 실행된다. 이 과정에서 전역 변수와 전역 함수가 생성되고, 이는 전역 실행 컨텍스트가 관리하는 전역 스코프에 등록된다.

특히 var 키워드로 선언된 전역 변수와 함수 선언문으로 정의된 전역 함수는 전역 객체의 프로퍼티와 메서드가 된다.

```jsx
 // 이 단계에서 실행되는 코드
 const x;

 function foo(a) { }
```

![](https://images.velog.io/images/ooooorobo/post/3411304b-0f16-42ee-bec1-cbb989648a51/image.png)

 2. 전역 코드 실행 단계

전역 코드의 평가가 끝나면 전역에 있는 코드가 한 줄씩 실행된다. 선언문은 평가 단계에서 실행되었으므로 이 단계에서는 실행되지 않는다. 이 단계에서는 값의 할당과 함수의 호출이 실행된다. **함수가 실행되면 실행된 지점에서 코드의 실행을 멈추고 함수 내부로 진입한다**.

 ```jsx
 // 이 단계에서 실행되는 코드
 x = 2;

 foo(123); // 함수 내부로 진입
 -----------
 console.log(1234); // 메소드 내부로 진입
```

![](https://images.velog.io/images/ooooorobo/post/61aa5d8c-e6e8-4272-bf05-82239f05241f/image.png)

 3. 함수 코드 평가 단계

함수가 호출되면 그 함수 몸체 부분으로 코드 실행 순서가 변경된다. 함수 내부 코드를 실행하기 전에, 마찬가지로 평가 단계를 거치며 매개변수나 지역 변수 선언문을 실행하여 매개변수와 변수를 생성한다. 매개변수와 지역 변수는 지역 스코프에 등록된다.

이 시점에 arguments 객체의 생성과 this 바인딩 결정도 이루어진다.

 ```jsx
 // 이 단계에서 실행되는 코드
 function foo(a) {
   const x;
 }
```

![](https://images.velog.io/images/ooooorobo/post/d782e219-425f-4ef9-b953-cf5ddbf348c5/image.png)

 4. 함수 코드 실행 단계

이 단계에서 foo 함수 내부의 코드가 순차적으로 실행된다. 실행되는 코드는 아래와 같다.

 ```jsx
 function foo(a) {
   x = 3;
   console.log(a);
 }
```

x에 값을 할당한 후, `console.log` 메서드를 호출한다. 메서드를 호출하기 위해서는 console 객체가 무엇인지를 알아야 한다. 따라서 스코프 체인을 통해 검색을 수행한다.

foo 함수에 console 객체가 존재하지 않으므로 상위 스코프를 검색한다. 그런데 console 객체는 전역 스코프에 따로 선언한 객체가 아니고, 전역 객체의 프로퍼티로 존재하는 객체이다. 이 객체를 스코프 체인을 통해 접근하기 위해서는 **전역 객체의 프로퍼티가 전역 스코프를 통해 검색 가능해야 한다.**

console 객체를 찾았으면, `log` 프로퍼티를 찾기 위해 console 객체의 **프로토타입 체인을 검색**한다. 프로토타입 체인 상에서 `log` 프로퍼티를 찾았으면, 해당 프로퍼티가 가지고 있는 함수에 a라는 인수를 넘겨주며 `console.log` 메서드를 실행한다. 메서드도 함수이므로 foo 함수 코드의 실행을 일시 중단하고 3번 단계로 돌아가 `console.log` 함수의 평가와 실행 과정을 수행한다.

모든 함수 코드의 실행이 끝났으면 다시 일시 중단되었던 시점의 전역 코드로 돌아가 남은 코드를 계속 실행한다.


이처럼 코드가 실행되기 위해서는,

1. **식별자**와 바인딩된 값을 스코프로 구분하여 관리되어야 한다.
2. 중첩 관계에 의해 **스코프 체인**을 형성하여 식별자를 검색할 수 있어야 한다.
3. 전역 객체의 프로퍼티는 전역 변수처럼 검색할 수 있어야 한다.

즉, **스코프, 식별자, 코드 실행 순서**의 관리가 필요하다. 이들을 관리하는 것이 **실행 컨텍스트**이다. 실행 컨텍스트는 **소스코드를 실행하는 데 필요한 환경을 제공하고 실행 결과를 실제로 관리하는 영역**이다.

즉 실행 컨텍스트는 스코프의 구현이고, 코드 실행 순서 관리를 구현한 내부 매커니즘이다.

식별자와 스코프는 실행 컨텍스트 **렉시컬 환경**으로, 코드 실행 순서는 **실행 컨텍스트 스택**으로 관리된다.

## 실행 컨텍스트 스택

실행 컨텍스트는 스택 자료구조로 관리된다. 이를 실행 컨텍스트 스택이라고 부른다. 이를 사용해 코드의 실행 순서를 관리한다.

소스코드가 평가되면 실행 컨텍스트가 생성되고, 실행 컨텍스트 스택 최상위에 쌓인다. 스택의 최상위에 있는 실행 컨텍스트는 현재 실행 중인 코드의 실행 컨텍스트이다. 이를 실행 중인 실행 컨텍스트라고 부른다.

## 렉시컬 환경

렉시컬 환경은 식별자와 식별자에 바인딩된 값, 상위 스코프 참조를 기록하는 자료구조이다. 렉시컬 환경에 식별자와 스코프에 대한 정보가 저장되며, 이를 통해 스코프와 식별자를 관리한다. 렉시컬 환경은 실행 컨텍스트에 포함되는 컴포넌트이다.

렉시컬 환경, 즉 스코프는 객체 형태로 생성되며, 식별자를 키로 등록하고 그 값에 바인딩된 값이 관리된다.

> 실행 컨텍스트는 `LexicalEnvironment` 컴포넌트와 `VariableEnvironment` 컴포넌트로 구성된다. 두 컴포넌트는 동일한 렉시컬 환경 객체를 참조한다. 이후에 상황에 따라 `VariableEnvironment` 컴포넌트를 위한 렉시컬 환경이 생성되어 두 컴포넌트의 내용이 달라질 수 있다. 이후 내용에서는 두 컴포넌트를 구분하지 않고 렉시컬 환경으로 통일해서 설명한다.

렉시컬 환경은 `EnvironmentRecord`, `OuterLexicalEnvironmentReference` (외부 렉시컬 환경에 대한 참조) 두 가지 컴포넌트를 가진다.

- `EnvironmentRecord`: 스코프의 **식별자를 등록하고 바인딩된 값을 관리**한다.
- `OuterLexicalEnvironmentReference`: 외부 렉시컬 환경에 대한 참조, **상위 스코프**를 가리킨다. 즉, 해당 실행 컨텍스트를 생성한 소스코드가 포함된 상위 코드의 렉시컬 환경이다. 이 컴포넌트를 통해 단방향 링크드 리스트 형태의 스코프 체인이 형성된다.

## 실행 컨텍스트의 생성 과 식별자 검색 과정

1. 전역 객체가 생성됨

다른 코드가 평가되기 이전에 생성된다. 빌트인 전역 프로퍼티와 빌트인 전역 함수, 표준 빌트인 객체와 호스트 객체가 생성된다.

2. 전역 코드 평가
    a. 전역 실행 컨텍스트 생성 `Global Execution Context`

![](https://images.velog.io/images/ooooorobo/post/650262c1-6b54-4866-8adf-5c65c0d88c29/image.png)

    b. 전역 렉시컬 환경 생성 - `Global Lexical Environment`

![](https://images.velog.io/images/ooooorobo/post/1d76e284-6df6-4304-8ebd-57b628eb6c01/image.png)

    c. 전역 환경 레코드 생성 - `Object Lexical Environment`

ES6 이후부터 변수를 선언하는 키워드는 var, let, const 세 가지가 있다. var로 전역에 변수를 선언하면 그 변수는 전역 객체의 프로퍼티가 되지만, let과 const는 그렇지 않다. 따라서 var로 선언된 변수와 let, const로 선언된 변수는 구분하여 관리되어야 한다. 이를 위해 **전역 환경 레코드는 객체 환경 레코드와 선언적 환경 레코드로 구성되어 있다.**

    d. 객체 환경 레코드 생성 - `Object Environment Record`

객체 환경 레코드는 `BindingObject` 라는 객체와 연결되는데, **이 객체는 맨 처음 생성했던 전역 객체**이다. var 키워드로 선언한 전역 변수와 함수 선언문으로 정의된 전역 함수는 `Object Environment Record.BindingObject` 객체, 즉 전역 객체의 프로퍼티와 메서드가 된다.

var 키워드로 선언한 변수는 선언과 초기화가 동시에 진행되어 `undefined` 값이 할당되어 있다. 즉, var 키워드로 전역에 변수를 선언하면 BindingObject에 변수 식별자를 프로퍼티 키로 등록하고, 그 값으로 undefined를 바인딩한다. 그렇기 때문에 변수 선언문 이전에도 변수에 접근할 수 있고, 즉 **호이스팅이 발생**한다.

    e. 선언적 환경 레코드 생성 - `Declarative Environment Record`

let, const 키워드로 선언한 전역 변수가 관리되는 곳이다. 이들은 전역 객체의 프로퍼티가 되지 않는다. 그리고 선언 단계만 수행되므로 변수에 바인딩된 값이 초기화되지 않은 상태로 있어 선언문 이전에는 변수에 접근할 수 없다. 이러한 현상을 일시적 사각지대(TDZ)라고 부른다. let, const로 변수를 선언해도 호이스팅은 발생하지만 일시적 사각지대로 인해 변수 선언문 이전에는 변수를 참조할 수 없다.


        ![](https://images.velog.io/images/ooooorobo/post/5963b8a7-5954-421e-8d82-5465ba164b81/image.png)

    f. this 바인딩

        `Global Lexical Environment.GlobalEnvironmentRecord` 의 `[[GlobalThisValue]]` 내부 슬롯에 this가 바인딩된다. 전역 코드에서 this는 전역 객체를 가리키므로, 전역 객체가 바인딩된다. this 바인딩은 전역 환경 레코드와 함수 환경 레코드에만 존재한다. this를 참조하면 이 환경 레코드의 this 내부 슬롯에 바인딩된 값을 참조하게 된다.

    g. 외부 렉시컬 환경에 대한 참조 결정 - `OuterLexicalEnvironmentReference`

        외부 렉시컬 환경에 대한 참조란, 현재 평가 중인 소스코드의 외부 소스코드 렉시컬 환경, 즉 상위 스코프이다. 이 값으로 스코프 체인을 구성한다. 전역 코드의 외부 소스코드는 없으므로 null이 할당된다. 즉, 전역 렉시컬 환경은 스코프 체인의 종점에 존재한다.

        ![](https://images.velog.io/images/ooooorobo/post/dd64f6b1-8246-4684-b2fd-6f426f64890a/image.png)

3 . 전역 코드 실행

전역 코드가 순차적으로 실행된다. 이 때 변수 할당문이나 함수 호출문을 만나게 되면, 해당 변수나 함수 이름이 선언된 식별자인지, 선언되었다면 어느 스코프의 식별자를 참조해야 하는지 확인하고 결정해야 한다. 이를 식별자 결정이라고 한다.

 - 식별자 결정은 현재 실행 중인 실행 컨텍스트에서 시작한다
 - 식별자 결정을 위해 스코프 체인을 따라 렉시컬 환경의 식별자를 검색한다
 - 만약 식별자를 찾지 못했다면 외부 렉시컬 환경에 대한 참조, 즉 상위 스코프로 이동해서 식별자를 검색한다
 - 전역 렉시컬 환경에서도 식별자를 찾을 수 없으면 참조 에러를 발생시킨다

4 . foo 함수 코드 평가

foo 함수가 호출되면 함수 코드의 평가 과정이 실행된다.

![](https://images.velog.io/images/ooooorobo/post/6e7c76a2-bc6f-4de8-a067-9c426eefda9d/image.png)

    a. 함수 실행 컨텍스트 생성 - `foo Execution Context`

빈 함수 실행 컨텍스트를 생성하고, 실행 컨텍스트 스택의 `Global Execution Context` 위로 쌓는다.

     b. 함수 렉시컬 환경 생성 - `foo Lexical Environment`
     c. 함수 환경 레코드 생성 - `FucntionEnvironmentRecord`

이 레코드에서 매개변수, arguments 객체, 함수 몸체에서 선언된 식별자를 관리한다.

     d. this 바인딩 - `FunctionEnvironmentRecord`의 `[[ThisValue]]` 내부 슬롯에 바인딩

함수 호출 방식에 따라 다르게 바인딩된다.

     e. 외부 렉시컬 환경에 대한 참조 결정 - `OuterLexicalEnvironmentReference`

여기에는 **해당 함수의 정의가 평가된 시점에 실행 중인 실행 컨텍스트의 렉시컬 환경 참조가 할당**된다. **렉시컬 스코프에서, 상위 스코프는 함수를 어디에 정의했는지에 따라 결정**된다. 함수 객체는 자신이 정의된 스코프, 즉 상위 스코프를 기억한다.

외부 렉시컬 환경의 참조는 함수 객체의 내부 슬롯 `[[Environment]]`에 저장되어 있다. `[[Environment]]`의 값은 함수 정의를 평가하여 함수 객체가 생성될 때 할당된다.

8. bar 함수 코드 실행 종료

더는 실행할 코드가 없어지면, 실행 컨텍스트 스택에서 bar 함수 실행 컨텍스트가 팝되어 제거되고, 실행 중인 실행 컨텍스트는 foo 실행 컨텍스트가 된다. bar 함수 실행 컨텍스트가 제거되어도 bar 함수 렉시컬 환경은 즉시 소멸되지는 않고, 가비지 컬렉터에 의해 메모리 공간 확보가 해제되어야 소멸한다. 누군가 bar 함수 렉시컬 환경을 참조하고 있다면 소멸하지 않는다.

## 블록 레벨 스코프

let, const로 선언한 변수는 블록 레벨 스코프를 가진다. 블록 레벨 스코프를 구현하기 위해서는 블록 단위로 렉시컬 환경을 생성해 주어야 한다. if, for문 등의 블록 레벨 스코프를 생성하는 모든 블록문이 실행되면 블록 렉시컬 환경을 생성하고, 현재 실행 컨텍스트의 렉시컬 환경은 블록 렉시컬 환경을 참조하고, 블록 렉시컬 환경은 `OuterLexicalEnvironmentReference`를 통해 기존 렉시컬 환경을 참조한다.
